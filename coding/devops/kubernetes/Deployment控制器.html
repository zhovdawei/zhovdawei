# Deployment控制器

# Pod与controllers的关系

• controllers：在集群上管理和运行容器的对象
• 通过label-selector相关联(pod模板文件中有配置相关参数)
• Pod通过控制器实现应用的运维，如伸缩，滚动升级等

控制器的另一个名称：工作负载(workload)

# Deployment功能与应用场景

• 部署无状态应用
• 管理Pod和ReplicaSet(也是一个控制器)
• 具有上线部署、副本设定、滚动升级、回滚等功能
• 提供申明式更新，例如只更新一个新的Image
应用场景：Web服务，微服务

# 使用Deployment部署一个应用

# 记住下面这个就可以了，根据提示写
$ kubectl create deployment --help

# 生成一个标准的yaml模板
[root@k8s-master ~]# kubectl create deployment web --image=nginx --dry-run -o yaml > web.yaml

# 然后可以编辑这个模板，配置我们真正想要的镜像，副本数啊，资源限制...等等参数
#################################################################
apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: web
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - image: david0607/java-demo
        name: java
#################################################################
# 执行
$ kubectl apply -f web.yaml

$ kubectl get pods

# 升级与回滚

# 1.创建

# 命令行部署不利于维护，一般输出yaml
kubectl create deployment web --image=nginx:1.14
kubectl get deploy,pods

# 2.发布

# service方式暴露出去
# --name=web 指定service名称，不指定默认和deployment的名称(这里是web)一致
# --port=80 service的端口，k8s集群内部使用
# --target-port=8080 容器应用的端口(这个应用是tomcat,一般默认是8080)
# --type=NodePort nodeport类型会分配一个随机的端口给到集群外部访问
kubectl expose --name=web deployment web --port=80 --target-port=8080 --type=NodePort 

# yaml文件方式暴露
# --dry--run 不提交到集群，检查是否可运行
# -o yaml 输出yaml文件格式
# > web-sercice.yaml 输出重定向到名为web-service.yaml的文件
kubectl expose --name=web deployment web --port=80 --target-port=8080 --type=NodePort --dry-run -o yaml > web-service.yaml

# 查看所有service
kubectl get service

# 3.升级

# 更新镜像版本
# 滚动升级，原来有三个容器，升级后起三个新容器，每起好一个新容器就移除一个旧容器
kubectl set image deployment/web nginx=nginx:1.15

# 查看升级
kubectl rollout status deployment/nginx-deployment

# 在线编辑deployment,即时生效
kubectl edit deployment/web

# 在线编辑service,即时生效

到现在我们应该能清楚，一个应用的运行，就是一个deployment的执行。而 一个deployment每次执行都伴随着一个replicas的存在。replicas作为deployment 的伴生产物，其主要有两个作用：

1.记录deployment的历史版本，回滚时需要从中指定版本或者回滚上一个。 如上面的输出，web有两个replicas,其中一个就是历史版本。

2.控制deployment的副本数,也就是pod的个数。比如一次版本更新，原先有三个副本， 那么replicas会起一个新版本的副本pod-就绪状态,然后删除一个旧版本的副本。 重复操作三次，直到副本数控制到3，且为新版本副本实例。service再把流量 切换到这三个新副本上，此时副本实例的状态才变为Running。
