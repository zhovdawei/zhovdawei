# Ingress对外暴露应用

# Pod与Ingress的关系

• 通过Service关联Pod
• 基于域名访问
• 通过Ingress Controller实现Pod的负载均衡 - 支持TCP/UDP 4层和HTTP 7层

# Ingress工作原理


1.ingress会监听master的变化，当有的service创建/删除/更新， ingress会拉取最新内容，更新自己的域名与service的配置数据。
2.ingress接收请求，解析域名(可能还有路由规则)映射的service, 将请求转发到service上，再由service自己默认的iptable(或lvs) 转发到具体的pod上 3.ingress具有nginx完整的功能，其实就是用nginx去实现的

# 创建Ingress规则

# 创建一个ingress.yaml文件，复制进去以下内容
###########################################################
# 资源的版本
apiVersion: extensions/v1beta1
# 资源的类型
kind: Ingress
# ingress的属性
metadata:
  name: web
spec:
  # 负载均衡的策略
  rules:
  # 配置域名 test.web.com
  - host: test.web.com
    http:
      # 路由配置类似/abc，也可以省略
      paths:
      # 域名+路由映射到 名为web端口为80的service上
      - backend:
          serviceName: web
          servicePort: 80
~                            
###########################################################

# 执行ingress规则配置文件
[root@k8s-master ~]# kubectl apply -f ingress.yaml 
ingress.extensions/web created

# 查看ingress资源配置
[root@k8s-master ~]# kubectl get ingress
NAME   HOSTS          ADDRESS   PORTS   AGE
web    test.web.com             80      49s

# 本地浏览器测试

访问前，我们需要在访问主机上配域名解析。
首先，上面运行ingress控制器，我们已经知道。ingress在两台子节点上都有pod 运行。用户首先是访问到ingress的pod,然后再由ingress做域名解析出service， 然后从service映射的多个pod中挑出一个转发过去。
Ingress控制器，默认是在每台node上起一个pod,当然你可以指定一批node 单独跑ingress,做负载均衡用。你只需要从这些pod中挑出一个，绑定域名映射即可。 用户的请求到了这个node上ingress,剩下的事就由ingress自己处理了。
我们这里就在本地hosts里面配置：192.168.163.109 test.web.com

# 扩展测试

以上测试，web的service其实还是NodePort类型。也就是说，它既可以通过node 的IP+端口访问，也可通过后面配置的ingress域名去访问。我们现在把service 设置成ClusterIP类型，让其只拥有集群内部访问的规则。然后再通过ingress 的域名去访问，看看这种情况下还能否生效。

# 删除原有的名为web的service
[root@k8s-master ~]# kubectl delete svc/web
service "web" deleted

# 创建类型为ClusterIP的service,还命名为web
[root@k8s-master ~]# kubectl expose --name=web deployment web --port=80 --target-port=8080 --type=ClusterIP
service/web exposed

# 查看service
[root@k8s-master ~]# kubectl get svc

我们再去通过node的ip+端口去访问肯定是不行了，不是NodePort，端口 也就不分配了。然后再通过域名访问，还可以正常访问。这就说明，ingress 将请求分发到servcie上属于集群内部的访问。它不需要具体的业务pod拥有 直接对外的暴露的权限。
