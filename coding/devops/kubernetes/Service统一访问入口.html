# Service统一访问入口

# Service存在的意义

• 防止Pod失联（服务发现）
• 定义一组Pod的访问策略（负载均衡）

# Pod与Servcie的关系

• 通过label-selector相关联 • 通过Service实现Pod的负载均衡（ TCP/UDP 4层） 

apiVersion: v1
kind: Service
metadata:
  creationTimestamp: "2019-12-10T07:49:37Z"
  ##########################################################
  # 这是一个标签只有一个键值对 app: web
  # 可以写多个，区分唯一性
  labels:
    app: web
  ##########################################################
  name: web
  namespace: default
  resourceVersion: "152754"
  selfLink: /api/v1/namespaces/default/services/web
  uid: 5ad1889d-094d-4f96-a4ac-4499be6b8ce7
spec:
  clusterIP: 10.1.150.171
  externalTrafficPolicy: Cluster
  ports:
  - nodePort: 32507
    port: 80
    protocol: TCP
    targetPort: 8080
  ##########################################################
  # 这是一个选择器，它只有一个键值对app: web
  # 这个选择器指向一个deployment:web
  # 如果deployment标签是多个，选择器也可以写多个
  # 确保唯一性即可
  selector:
    app: web
  ##########################################################
  sessionAffinity: None
  type: NodePort
status:
  loadBalancer: {}

# Service几种类型

# ClusterIP 类型

分配一个内部集群IP地址，只能在集群内部访问（同Namespace内的Pod）， 默认ServiceType。ClusterIP 模式的 Service 为你提供的，就是一个 Pod 的稳定的 IP 地址，即 VIP。

# NodePort 类型

分配一个内部集群IP地址，并在每个node上创建一个端口，用node的ip+端 口访问这台node主机上的一组pod,继而再转发到具体的某个pod。很明显， 这样指定IP+端口的访问方式，会造成某个节点的大量访问。一般的，都是 在节点外加一层LB做负载均衡。用户访问LB，由LB去分发到具体的节点上， 继而分发的具体的pod上。

# LoadBalancer 类型

主要是公有云平台在使用。分配一个内部集群IP地址，并在每个节点上启 用一个端口来暴露服务。除此之外，Kubernetes会请求底层云平台上的负 载均衡器，将每个Node（[NodeIP]:[NodePort]）作为后端添加进去。

# 创建service

# 命令行创建
# --name=web 指定service名称，不指定默认和deployment的名称(这里是web)一致
# --port=80 service的端口，k8s集群内部使用
# --target-port=8080 容器应用的端口(这个应用是tomcat,一般默认是8080)
# --type=NodePort nodeport类型会分配一个随机的端口给到集群外部访问
kubectl expose --name=web deployment web --port=80 --target-port=8080 --type=NodePort 

# yaml文件方式创建
# --dry--run 不提交到集群，检查是否可运行
# -o yaml 输出yaml文件格式
# > web-sercice.yaml 输出重定向到名为web-service.yaml的文件
kubectl expose --name=web deployment web --port=80 --target-port=8080 --type=NodePort --dry-run -o yaml > web-service.yaml

# 查看所有service
kubectl get service

# service转发到pod

service转发到pod上，默认是使用的iptables实现的负载均衡。iptables 内部维护一张规则表，当规则越来越多时，pod频繁变更时，iptables从表中 匹配规则的效率会降低。(k8s后面引入了lvs，专门负责大并发的负载均衡)
