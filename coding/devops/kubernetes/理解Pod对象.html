# 理解Pod对象

# Pod基本概念

最小部署单元
一组容器的集合
一个Pod中的容器共享网络命名空间与存储
Pod是短暂的

# Pod存在的意义

Pod为亲密性应用而存在。
亲密性应用场景：
两个应用之间发生文件交互
两个应用需要用过127.0.0.1或者socket通信
两个应用需要发生频繁的调用

# 查看一下 my-pod分布在哪个节点上
[root@k8s-master ~]# kubectl get pods -o wide
NAME                         READY   STATUS    RESTARTS   AGE     IP            NODE        NOMINATED NODE   READINESS GATES

4a503d9ee291        nginx                                               "nginx -g 'daemon of…"   19 minutes ago      Up 19 minutes                           k8s_nginx_my-pod_default_bdac93e9-8908-4da7-9657-080f48ef5d12_0

e6c2f867bdb6        registry.aliyuncs.com/google_containers/pause:3.1   "/pause"                 19 minutes ago      Up 19 minutes                           k8s_POD_my-pod_default_bdac93e9-8908-4da7-9657-080f48ef5d12_0

################################ 结论 ################################
docker网络管理中，容器的网络可以指定另一个容器的网络，然后多个容器共享一个网络。
这里的pause容器就是用来创建网络约束的，java和nginx都加入它，三者共享一个网络。
pause是pod默认创建的容器。

# Pod存储共享

pod从崩溃到恢复，如果数据没有持久化，恢复后的pod就是一个新生儿。
这显然不是我们希望的，我们希望数据得到持久化。
需要持久化的数据：
1.临时数据
2.日志
3.业务数据
如果pod内的多个容器是亲和性高的，比如一个生产数据，一个读取数据，那么就需要做数据共享。

# 可以执行下面这个pod模板，验证数据共享
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
  - name: write
    image: centos
    command: ["bash","-c","for i in {1..100};do echo $i >> /data/hello;sleep 1;done"]
    volumeMounts:
      - name: data
        mountPath: /data
  - name: read
    image: centos
    command: ["bash","-c","tail -f /data/hello"]
    volumeMounts:
    - name: data
      mountPath: /data
volumes:
- name: data
  emptyDir: {}
  
# 上面的数据共享，是两个容器的/data目录挂载到pod所在节点的一个emptyDir类型的数据卷上，从而实现数据的共享
# emptyDir 是一个空目录，跟随pod的生命周期
# pod崩溃不是死亡，emptyDir依然存在，pod恢复后，容器可以从emptyDir恢复数据
# pod移除，emptyDir也随之丢失

# 判断是否准备完成，否则重建恢复

  tcpSocket:
    port: 8010
  initialDelaySeconds: 60
  periodSeconds: 10

  tcpSocket:
    port: 8010
  initialDelaySeconds: 60
