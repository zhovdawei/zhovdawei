# 有一个问题


貌似第一次远程访问，这个yes必须手动输入。jenkins没办法自己绕过这一关， 即使是ssh方式第一次也会出现这个问题。如果各位也遇到了这个诡异的问题， 自己寻找解决办法，实在不行，就手动操作一发吧。

# 三种参数化构建方式举例

这里我们以git拉取分支代码场景，来演示动态配置分支这个参数的几种方式。 下面我们先创建一个任务test。

# 字符参数构建


上面这个演示分三步。
第一步，创建了一个字符类型的参数：branch,然后设置了一个默认值：master
第二步，在git配置栏，分支的取值来至于 ${branch}
第三步，配置了个脚本执行，打印branch的值
当我们再构建的时候，可以看到bulid with parameter。点击进去，就可以看到， 需要我们填入一个值。jenkins服务器从远端拉取代码时，就会把填入的值当做一个 分支。如果不填，那就是默认master。构建完，是成功的，我们从控制台的 日志打印也看到了branch的值。 这样明显不够优雅，如果你的分支太多太长，你还记住的吗？就算你的记得住， 你填的对吗？所以如果是选择分支，就太好了。下面我们演示使用参数插件， 来生成参数化构建的选择方式。

# Extended Choice Parameter

分两种方式测试：
Value方式测试
PropertyFile方式测试

# 安装插件

在插件管理页面，可选插件搜索“Extended Choice Parameter”。选择它，然后直接安装。 等待安装完成再进行下一步。

# Value方式测试


注意Extended Choice Parameter几个选项，对于分支的选择，我们要注意是
Parameter Type : 勾选单项选择。虽然它默认是单选，但是我们需要注意到。
Delimiter : 分隔符，注意中英文逗号。
Value : 勾选并在下方填写多个选项值，英文逗号隔开。
再次进入构建操作，发现，现在是选择分支了。这就比较人性化了。

下面用property演示。

虽然上面两种方式给与了很大的便利性，但是对于git来说，还是不够优雅。 但凡开发者，增加删减了分支，我们对于branch的参数必须重新配置。下面 我们用另外一个插件，它能自动获取分支的变更，提供给我选择最新的分支 版本。这就很优雅了。嘿嘿。
