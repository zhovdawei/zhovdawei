# Jenkins与k8s的自动化CI&CD流水线实战

# 目标流程


一般企业就是上面的流程。
开发：提交代码，推送到Git仓库。
测试：操作Jenkins的流水线任务，完成测试环境发布，验证测试程序。
运维：测试提交镜像号，运维发布生产环境。
用户：用户访问统一LB入口，由LB负载到某个pod上的具体容器，即目标应用程序。
我们这里只需要实现部署到测试环境即可，如果这一次构建的镜像是通过了测试验证。 发布生产，只需要在生产K8S集群的Dashboard上完成操作就行。这个里面最重要还是 得到一个满足需求的镜像。

# 部署K8S集群

stage('部署') {
    sh '''
        REPOSITORY=depo.zhoudawei.com/mytest/java-web-1:${branch}
        ssh root@192.168.163.108 "/usr/bin/rm -rf k8s-java |true"
        ssh root@192.168.163.108 "/usr/bin/mkdir k8s-java"
        ssh root@192.168.163.108 "/usr/bin/cd k8s-java"
        
        # 依次关闭ingress,service,deployment
        # 重复构建可以利用deployment升级来替换，我这里为了省事就干掉原来的
        ssh root@192.168.163.108 "/usr/bin/kubectl delete ingress/k8s-java"
        ssh root@192.168.163.108 "/usr/bin/kubectl delete svc/k8s-java"
        ssh root@192.168.163.108 "/usr/bin/kubectl delete deploy/k8s-java"
        
        # 复制deployment模板，并替换其中的变量值
        ssh root@192.168.163.108 "/usr/bin/cp /root/deploy-demo.yaml deploy.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's/param-name/k8s-java/g' deploy.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's/param-num/3/g' deploy.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's!param-image!$REPOSITORY!g' deploy.yaml"
       
        # 复制service模板，并替换其中的变量值
        ssh root@192.168.163.108 "/usr/bin/cp /root/svc-demo.yaml svc.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's/param-name/k8s-java/g' svc.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's/param-port/80/g' svc.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's/param-target-port/8080/g' svc.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's/param-type/ClusterIP/g' svc.yaml"
        
        # 复制ingress模板，并替换其中的变量值
        ssh root@192.168.163.108 "/usr/bin/cp /root/ingress-demo.yaml ingress.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's/param-name/k8s-java/g' ingress.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's/param-url/k8s.test.com/g' ingress.yaml"
        ssh root@192.168.163.108 "/usr/bin/sed -i 's/param-port/80/g' ingress.yaml"
        
        # 依次启动deployment,service,ingress
        ssh root@192.168.163.108 "/usr/bin/kubectl apply -f deploy.yaml"
        ssh root@192.168.163.108 "/usr/bin/kubectl apply -f svc.yaml"
        ssh root@192.168.163.108 "/usr/bin/kubectl apply -f ingress.yaml"
        
    '''  
}

# 构建一条龙


如上演示，新建一个k8s-java的流水线任务。前面三步没啥可说的。前一篇都说过了。
然后这里需要注意的是部署这个环节的脚本，我写了较多重复参数，优化起来是可以在脚本前面部分， 把相关参数设置好，后面stage里面的脚本完全可以固定模式，引用前面的变量即可。这样一来，如果 有新项目要创建人物，我们可以要求他们按照要求填写项目的Git地址，项目名...等参数。这样就可以 去完成自动化的构建任务了。
