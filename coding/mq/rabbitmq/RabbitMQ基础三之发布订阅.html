    # rabbitmq消息传递模型

    RabbitMQ消息传递模型中的核心思想是生产者从不将任何消息直接发送到队列。 实际上，生产者经常甚至根本不知道是否将消息传递到任何队列。 相反，生产者只能将消息发送到交换机。交流是一件非常简单的事情。 一方面，它接收来自生产者的消息，另一方面，将它们推入队列。交换机必须 确切知道如何处理收到的消息。--- 官方原文 (opens new window)
    有几种交换类型可用：direct，topic，headers 和fanout。我们将集中讨论最后一个- fanout(扇出)。
    ❶疑问：消息是否必须由exchange推到队列
    我们在前面简单队列和工作队列中，都貌似直接将生产的消息发送到了队列，然后消费者 监听队列，拿到消息去消费。这里不妨跟踪一下源码：
    从简单队列生产消息的方法进去：

    # 消费者一改造

    ❶新增一个配置类：
    定义了一个队列和一个交换机，定义一个绑定，将队列绑定到该交换机上。 这个交换机就是生产者中消息的发送的目的地，经由它将消息推到绑定的队列上去。

    @Bean

        return new Queue("consumerSubQueue");
    }

    @Bean
    public Binding bindingAtPsExchange(Queue consumerSubQueue,FanoutExchange psExchange){
        return BindingBuilder.bind(consumerSubQueue).to(psExchange);

    # 消费者二改造

    ❶新增配置类：
    这里面定义的队列与之前的不同：AnonymousQueue。它是一个临时队列，当客户端 (消费者端)与服务端(rabbitmq-server)断开连接时，该队列自动删除并且解除与交 换机的绑定。之后我们测试一下。

    ❷新增队列监听类
    consumerOneSubQueue是定义的一个临时队列，再其配置生效前，我们并不知道它的name, 所以我们借由它的属性值，动态获取name值。

    package com.david.mq;

import org.springframework.amqp.core.Queue;
import org.springframework.amqp.rabbit.annotation.RabbitListener;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

/**
 * @author zhoudawei
 * @mail zdw0607@gmail.com
 * @date 2019-11-07 17:57
 */
@Component
public class SubQueueReceiver {

    @Autowired
    private Queue consumerOneSubQueue;

    @RabbitListener(queues = "#{consumerOneSubQueue.name}" )
    public void receive(String msg) {
        String queueName = consumerOneSubQueue.getName();
        System.out.println("接受者 => "+queueName+" => "+msg);
    }

}

    # 测试流程

    ❶先运行生产者测试方法，往psExchange里面丢十条消息

    ❷再运行两个消费者应用，然后可以发现，没有任何消息消费信息打印出来。这里就可以得出 一个结论。exchange交换机是不存储消息。当一个交换机没有绑定任何队列的时候，往该交换 机里发送消息，它不知道投递到哪个队列，消息就自动释放掉了。那么现在两个消费者定义的 队列随着应用启动区创建，并绑定到交换机上。如下图：consumerSubQueue是我们定义的一个 普通队列。而spring.gen-xxx则是我们定义的临时类，其名字是spring-amqp实现类库帮我们 生成的。

    ❸再次运行生产者测试方法
    现在我们就能看到，两个各自消费了10条消息。从而验证了消息的发布/订阅功能。

    ❹关闭两个消费者应用
    临时队列已经解除与psExchange的绑定，再去Queue管理页面看，临时队列也消失了(我这里 就不截图了，直接给结论)。思考一下这种情况，再往psExcchange发十条消息，消息会被投 递到consumerSubQueue队列中。当再次启动两个消费者，消费者一能消费到消息，而消费二 重新创建队列，然后再绑定在交换机，等到下次消息的投递。临时队列的使用，我们务必要 清除它的运作机制，以免出现消息丢失的情况。
